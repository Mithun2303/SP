Technical Debt

PPT 4

What is Technical Debt?

Technical debt is the debt that accrues when you knowingly or unknowingly make wrong or non-optimal design decisions. — Ward Cunningham (coined the metaphor)

Like financial debt — you take a shortcut (principal) and pay interest later through harder maintenance.

Code Debt — static analysis violations, inconsistent style Design Debt — design smells, rule violations Test Debt — lack of tests / coverage Documentation Debt — missing/outdated docs

Defect vs Technical Debt

  • Defects — visible to users, get immediate attention
  • Tech Debt — largely invisible, receives little attention

Components of Tech Debt

  • Principal — the original hack/shortcut
  • Interest — accumulated cost of not fixing it (compound in nature)

Impact on Cost of Change (CoC)

  • Well-maintained system: actual CoC ≈ optimal CoC
  • As tech debt increases → CoC increases
  • Extreme case → "Technical Bankruptcy" (CoC becomes prohibitively high)

What Causes Technical Debt?

Schedule Pressure Lack of skilled designer No design principles Unaware of design smells

How to Manage?

  • Increasing awareness of technical debt
  • Detecting and repaying technical debt (refactoring)
  • Prevent accumulation of technical debt

📦 Case Study 1: The 12-Year-Old Product

  • Medium company, 12-year-old monolithic client-server system under upgrade pressure
  • Debt found: No client/server separation, tight coupling, no interfaces, poor encapsulation
  • Resistance: Fear of breakage, no awareness of tech debt, original devs left
  • Strategy: Refactor one critical part; Boy Scout Rule — clean code as you add features
  • Outcome: 6-month delay, fewer features, but cleaner foundation for future (avoided bankruptcy)

⚡ Real Example: Knight Capital — $460 Million Bug (2012)

  • Debt: Deprecated 8-year-old "Power Peg" code was never deleted
  • Trigger: Deployment script failed to update one of 8 servers
  • Result: Old code activated → $10M loss/minute → company went from $1.5B valuation to acquired for $1.4M in 48 hours

Design Smells — Introduction

PPT 4 + Book

What Are Design Smells?

Structures in design that indicate violation of fundamental design principles and negatively impact design quality.
Structural Indicators (not bugs) Symptom, not Cause Maintainability Killers

Key Quality Attributes Impacted

Understandability Reliability Changeability Extensibility Reusability Testability

PHAME Classification Framework

Hierarchical classification based on Booch's object model. Maps every smell to the specific design principle it violates.

P — Principle of Abstraction

Simplification of entities — reduction and generalization

H — Principle of Hierarchy

Organizing types using meaningful classification

A — Principle of Abstraction (naming)

Two-word naming: Adjective + Principle

M — Modularization

Cohesive and loosely coupled abstractions

E — Encapsulation

Separation of interface and implementation

💡

Naming scheme: Adjective + Principle e.g. "Broken Modularization", "Deficient Encapsulation", "Missing Abstraction"

Causes of Design Smells

Principle Violation

E.g. java.util.Calendar violates SRP → Multifaceted Abstraction

Inappropriate Pattern Use

Patterns misapplied → over-engineering / Class Bloat

Language Limitations

Java generics don't support primitives → duplicate code → Unfactored Hierarchy

Procedural Thinking in OOP

Thinking "doing" not "being" → Imperative Abstraction, Missing Hierarchy

Viscosity

Software viscosity: correct fix takes too long; Environment viscosity: slow tools/processes

Non-Adherence to Best Practices

Leaving composite refactoring halfway → structural inconsistencies

Documenting a Smell (Template)

  • Name & Description
  • Rationale
  • Potential Causes
  • Example
  • Suggested Refactoring
  • Impacted Quality Attributes
  • Aliases
  • Practical Considerations

Abstraction Smells (7 Types)

PPT 4

3.1 Missing Abstraction PHAME: A

Clumps of data or encoded strings are used instead of creating a class or interface. Violates: crisp conceptual boundaries & unique identity.

Causes

  • Inadequate design analysis / tight deadlines
  • Lack of refactoring as requirements evolve
  • Misguided performance focus (arrays over objects)

Examples

  • ISBN stored as string → duplicated validation everywhere
  • Data Clumps: x1,y1,x2,y2 always together → needs a Rectangle class
  • Stack trace stored as string → clients depend on fixed format

Refactoring

  • Create a class when: validation is needed, constructors/formatting required, resource handling needed
  • Not every primitive needs abstraction — avoid over-engineering
↓ Understandability ↓ Changeability ↓ Reusability ↓ Reliability
Aliases: Primitive Obsession, Data Clumps

3.2 Imperative Abstraction PHAME: A

An operation is represented as a class instead of a method. Classes should represent entities, not actions.

Causes

  • Procedural thinking in OOP
  • One-method classes = actions not entities

Examples

  • Classes: CreateReport, CopyReport, DisplayReport → should all be methods in a Report class
  • Classes: OffsetCalculator, SaveHandler → merge into ImagePair class
💡

When acceptable (Reification): Command Pattern, Strategy Pattern, State Pattern — converting behavior to objects intentionally is fine

Alias: Operation Class

3.3 Incomplete Abstraction PHAME: A

Abstraction fails to provide complementary or interrelated methods, forcing clients to work around missing functionality.
  • Example: class has initialize() but no dispose()
  • Real Example (Java Swing): ButtonModel has setGroup() but no getGroup(). Fixed in JDK 1.3 by adding getGroup() to DefaultButtonModel.

Causes

  • Missing overall perspective — focused only on immediate needs
  • Language convention overlooked (e.g. missing hashCode() when equals() is defined in Java)
  • Architecture docs specify creation but omit deletion
Encapsulation Violation Reduced Cohesion SRP Violation

3.4 Multifaceted Abstraction PHAME: A

A single class is assigned multiple responsibilities. Violates: Single Responsibility Principle (SRP).

Causes

  • General-purpose names (Node, Component, Element) become catch-all
  • Evolution without periodic refactoring → "blobs"
  • Viscosity — avoiding creating a new class
  • Mixing domain logic with presentation

Real Example

  • java.util.Calendar — handles both date AND time: 2825 lines, 67 methods, 71 fields!
  • Led to creation of Joda-Time library
  • Java 8 introduced java.time as replacement

Refactoring — Extract Class

  • Identify logical clusters of methods/fields, apply "Extract Class"
  • Example: Image class (120 public methods, 50,000+ lines) → refactored into ImageLoader, ImageRenderer, ImageProcessor, ImageWriter

3.5 Unnecessary Abstraction PHAME: A

An abstraction that is not actually needed gets introduced — trivial or no meaningful responsibility. Violates YAGNI principle.

Causes

  • Procedural thinking — classes that "do" not "be"
  • Constant interfaces in Java (abuse of interface)
  • Over-engineering

Real Examples (JDK)

  • java.util.FormattableFlags — class holding 3 integer constants (should be enum)
  • javax.swing.WindowConstants — interface holding 4 constants (constant interface anti-pattern)
  • BestSellerBook class that only delegates all calls to Book — no unique behavior
↓ Understandability ↓ Reusability

3.6 Unutilized Abstraction PHAME: A

An abstraction is left unused — not directly used or not reachable.

Two Forms

  • Unreferenced abstractions — concrete classes used by nobody
  • Orphan abstractions — standalone interfaces/abstract classes with no derived abstractions

Causes

  • Speculative design (future-proofing)
  • Changing requirements (old abstraction no longer needed)
  • Leftover garbage during maintenance
  • Fear of breaking code

Real Examples

  • sun.misc.Service (JDK 1.3) → became unused after java.util.ServiceLoader (JDK 1.6). Removed in JDK 9.
  • Custom concurrent utilities became unused after java.util.concurrent arrived (JDK 1.5)
  • Visualization tool — report-generation hierarchy created speculatively, never implemented (Orphan)

Refactoring

  • Simplest fix: Delete the abstraction entirely
  • For public APIs: mark @Deprecated instead of removing immediately
Aliases: Unused classes, Speculative generality

3.7 Duplicate Abstraction PHAME: A

Two or more abstractions have identical names or identical implementation or both.

Two Forms

  • Identical Name — same names, may share behavior (confusing)
  • Identical Implementation — semantically identical member definitions not factored together (code clones)

Causes

  • Copy-paste programming / AI-generated code not understood
  • Ad hoc maintenance over years
  • Lack of communication between teams
  • Final class needs extension → duplicate created

Real Examples

  • Same Name: Three Timer classes in JDK — javax.swing.Timer, java.util.Timer, javax.management.timer.Timer. Same description → confusing for beginners.
  • Same Name: java.util.Date and java.sql.Date — same name in different packages → import ambiguity
  • Duplicate: Scanner class in eclipse.jdt.internal ≈ PublicScanner in eclipse.jdt.core — ~3500 lines identical

Refactoring

  • Same names → rename uniquely (e.g. javax.swing.Timer → EventTimer, javax.management.timer.Timer → AlarmTimer)
  • Same implementation → remove one, or factor common code into a parent class
  • java.sql.Date → rename to java.sql.SQLDate; convert inheritance to delegation (wrap java.util.Date)
Aliases: Alternative classes with different interfaces, Duplicate design artifacts

Encapsulation Smells (4 Types)

PPT 4

Principle of Encapsulation

Advocates separation of concerns — interface and implementation. Hide "how" and expose only "what".

Hide Implementation Details

Expose what the abstraction offers, not how it is implemented. Internal representation and algorithms should be hidden.

Hide Variations

Hide implementation variations in types or hierarchies. Makes changes easier without impacting clients.

4.1 Deficient Encapsulation PHAME: E

Declared accessibility of members is more permissive than actually required (e.g. public fields that should be private).

Causes

  • Easier testability — making private methods public
  • Procedural thinking — treating data as global variables
  • Quick-fix delivery pressure shortcuts

Examples

  • Point class — public x, y fields; should be private with getters/setters
  • JDK 7: 508 classes with at least one public field; PNGMetadata has 85+ public fields!
  • SizeRequirements.alignment — public, so alignment validation can't be enforced
  • System.out, System.err, System.in — public final fields (global variables) locked in forever

Refactoring for System.out Example

  • Introduce a Console abstraction wrapping stream details
  • Java 1.6 did this → java.io.Console with reader()/writer() methods
💡

Performance myth: Developers believe direct field access is faster — but JIT compilers inline accessor methods, making the difference negligible.

4.2 Leaky Encapsulation PHAME: E

Abstraction exposes or leaks implementation details through its public interface (methods reveal internal data structures or algorithms).

Causes

  • Lack of awareness of what should be hidden
  • Fine-grained interface exposing internal details

Examples

  • TodoList.getListEntries() returns LinkedList → exposes internal data structure; clients can mutate it; changing to ArrayList breaks clients
  • CustomListBox.bubbleSort() — method name reveals algorithm. If changed to quickSort, must rename and break clients OR keep misleading name

Refactoring

  • Return deep copy instead of reference
  • Return immutable object
  • Return base type: return List or Collection not LinkedList
  • Return Iterator if traversal is the goal
  • Rename bubbleSort() → sort() (algorithm-agnostic name)

4.3 Missing Encapsulation PHAME: E

Client is tightly coupled to different variations of a service. New variations cause class explosion. Violates Open-Closed Principle (OCP).

Causes

  • Lack of awareness of OCP
  • Lack of refactoring as requirements change
  • Mixing up concerns (orthogonal concerns merged)
  • Naive design: creating a class for every combination

Example (Encryption)

  • Encryption class with DES hard-coded
  • Add AES → class grows, hard to maintain
  • Add TDES + Image/Text content → class explosion
  • Fix: Strategy pattern — encapsulate algorithm variation separately
↓ Understandability ↓ Changeability ↓ Reusability

4.4 Unexploited Encapsulation PHAME: E

Client code uses explicit type checks (if-else/switch on object type) instead of exploiting polymorphism already encapsulated in a hierarchy.

Causes

  • Procedural thinking — switch/if for behavior selection
  • Lack of firm grasp of OOP principles

Example

  • ColorModel uses switch on DataBuffer types to call getPixel()/getSize()
  • Fix: define getPixel() and getSize() in DataBuffer hierarchy; reduce switch to two polymorphic calls
↓ Changeability ↓ Reusability ↓ Testability ↓ Reliability

Modularization Smells (4 Types)

PPT 5

Principle of Modularization

Create cohesive and loosely coupled abstractions through localization and decomposition.

Localize Related Data & Methods

Each class owns its data and the methods that operate on it.

Decompose to Manageable Size

Break large abstractions into smaller focused ones. Avoid fragmentation and bloat.

Create Acyclic Dependencies

Dependency graphs must be cycle-free. Cycles cause ripple effects.

Limit Dependencies (Fan-in & Fan-out)

Low fan-in = fewer clients break on change. Low fan-out = less exposure to others' changes.

1. Broken Modularization Violates: Localize related data & methods

Data and/or methods that should be in one abstraction are scattered across multiple abstractions → tight coupling.

Causes

  • Procedural thinking in OOP
  • Lack of knowledge of existing design

Example (DeviceData / Device)

  • DeviceData holds only fields (no methods)
  • Device holds all methods + a DeviceData reference
  • Data and behavior are split across 2 classes
// BEFORE (Broken)
class DeviceData {
    public String deviceId;
    public String devicePath;
    public boolean enabled;
}
class Device {
    DeviceData data; // owns data
    String getDevicePath(String id) {...}
    boolean isEnabled() {...}
}
// AFTER (Fixed)
class Device {
    private String deviceId;
    private String devicePath;
    private boolean enabled;

    public String getDevicePath() {...}
    public boolean setDevicePath(String p){...}
    public boolean isEnabled() {...}
}

Refactoring Techniques

  • Convert Procedural Design to Objects
  • Move Method — if Class A's method is used more by Class B
  • Move Field — if Class A's field is used more by Class B
↓ Understandability ↓ Changeability ↓ Reusability ↓ Reliability

2. Insufficient Modularization Violates: Decompose to manageable size

Abstraction has not been completely decomposed — too large or too complex.

Two Forms

  • Bloated Interface — too many members (methods/fields)
  • Bloated Implementation — too many methods or one method with excessive complexity

Causes

  • Providing centralized control
  • Creating large classes for multiple clients
  • Grouping all related functionality together

Examples

  • Bloated Implementation: java.net.SocketPermission — Cyclomatic Complexity 193; getMask() method alone has CC=81
  • Bloated Both: java.awt.Component — 332 methods, 11 nested/inner classes, 107 fields. Tries to handle Events, Layout, Focus, Fonts all in one class.

Refactoring

Bloated Interface

  • Extract Cohesive Subsets → separate class
  • Interface Segregation Principle (ISP)

Bloated Implementation

  • Simplify with helper functions
  • Encapsulate each responsibility separately

3. Cyclically-Dependent Modularization Violates: Acyclic dependencies

Two or more abstractions depend on each other — directly (A↔B) or indirectly (A→B→C→A).

Key Concepts

  • Cycle — start at abstraction A, return to A through dependencies
  • Tangle — multiple cycles, complex tightly coupled design
  • Ripple Effect — change in A cascades back to A
  • Modules must be understood, changed, tested, reused together

Causes

  • Improper responsibility realization
  • Passing self reference (this) instead of data
  • Poor call-back implementation
  • Hard-to-visualize indirect dependencies

Examples

  • Example 1: SecureDocument ↔ DESEncryption — SecureDocument calls encrypt(), passes "this", DESEncryption uses it to access content → cycle
  • Example 2: Image ↔ Encryption in medical app — Image passes "this" to encrypt(), Encryption accesses image content → cycle

Refactoring (Breaking Cycles)

1. Introduce Interface

DESEncryption implements IEncryption; SecureDocument depends on IEncryption not concrete class. AESEncryption can now be added without changing SecureDocument.

2. Remove Unnecessary Dependency

Eliminate avoidable dependency — use Move Method / Move Field

3. Move Code to New Abstraction

Move SecurityManager to handle interaction between Image and Encryption. Image no longer depends on Encryption.

4. Merge Abstractions

If both represent a single concept, combine into one class

Aliases: Dependency Cycles, Cyclic Dependencies, Bidirectional Relation, Cyclic Class Relationships

4. Hub-like Modularization Violates: Limit dependencies

An abstraction has too many incoming AND outgoing dependencies — looks like a hub.

Types of Dependencies

  • Incoming (Fan-in) — many classes depend on this abstraction
  • Outgoing (Fan-out) — this abstraction depends on many others

Causes

  • Improper responsibility assignment (too many roles)
  • "Dump-all" utility classes (e.g. SwingUtilities)

Example

  • java.awt.Font — used for rendering text as glyphs on Graphics, Component
  • Incoming: 19 classes depend on Font
  • Outgoing: 34 classes used by Font
  • Result: High coupling, complex to maintain

Refactoring

  • Split Responsibilities — divide class with multiple roles into smaller ones
  • Move Misplaced Members — shift methods/fields to appropriate abstractions
  • Chain of Responsibility Pattern — connect receivers in a chain, hub notifies only the first
Aliases: Bottlenecks, Local Hubs, Man-in-the-Middle

Hierarchy Smells (8 Types)

PPT 5

Enabling Techniques for Hierarchy

Apply Meaningful Classification

Identify commonalities & differences; classify supertypes and subtypes in levels

Apply Meaningful Generalization

Extract common behaviors from subtypes to reduce code duplication

Ensure Substitutability

Liskov Substitution Principle — supertype reference can be replaced with subtype object without altering behavior; enables dynamic polymorphism

Avoid Redundant Paths

Subclass should not have multiple ways to inherit from same superclass (Diamond problem)

Ensure Proper Ordering

Subtype knows about supertype, NOT the reverse. Reverse creates circular dependency.

1. Missing Hierarchy

Conditional logic or tagged type is used to manage variations explicitly instead of using a hierarchy (inheritance).

Causes

  • Procedural approach — C-style encoded types with conditionals
  • Misguided simplistic design — switch seems simpler
  • Overlooking inheritance as a design technique

Impact

  • ↓ Understandability — switch complexity
  • ↓ Extensibility — new type = change all switch statements
  • ↓ Reusability & Testability
  • ↓ Reliability — must check all types
// BEFORE: Switch smell
class Employee {
    enum Type { ENGINEER, MANAGER, SALESMAN }
    private Type type;
    public double calculateBonus() {
        switch (type) {
            case ENGINEER: return salary * 0.1;
            case MANAGER:  return salary * 0.2;
            case SALESMAN: return salary * 0.15;
        }
    }
}
// AFTER: Proper hierarchy
abstract class Employee {
    protected double salary;
    public abstract double calculateBonus();
}
class Engineer extends Employee {
    public double calculateBonus() { return salary * 0.1; }
}
class Manager extends Employee {
    public double calculateBonus() { return salary * 0.2; }
}

2. Unnecessary Hierarchy

Inheritance hierarchy is unnecessary — subclasses differ only in data, not behavior.

Causes

  • Subclasses instead of instantiating (differ only in data)
  • Taxonomy mania — overuse of inheritance

Impact

  • ↓ Understandability — complicates design
  • ↓ Extensibility — adding new types explodes hierarchy
  • ↓ Testability — more effort for unnecessary types
// BEFORE: Subclasses for data only
abstract class Coffee {
    public abstract String getName();
    public abstract double getPrice();
}
class Latte extends Coffee {
    public String getName() { return "Latte"; }
    public double getPrice() { return 3.50; }
}
class Espresso extends Coffee {
    public String getName() { return "Espresso"; }
    public double getPrice() { return 2.00; }
}
// AFTER: Use instances, not subclasses
class Coffee {
    private String name;
    private double price;
    public Coffee(String name, double price) {
        this.name = name; this.price = price;
    }
    public String getName() { return name; }
    public double getPrice() { return price; }
}
// Usage: just create instances!
Coffee latte = new Coffee("Latte", 3.50);
Coffee espresso = new Coffee("Espresso", 2.00);

3. Unfactored Hierarchy

Unnecessary duplication among types in a hierarchy. Violates DRY principle.

Two Types of Duplication

Type 1: Horizontal (Siblings)

Same code in parallel branches — CreditCardPayment and PayPalPayment both have identical validate() method

Type 2: Vertical (Super/Sub)

Redundant code between parent and child — child redefines what it already inherits. AbstractQueuedSynchronizer: 1,278 of 2,110 lines identical!

Three Forms

Form 1: Unfactored Interface

Same method signatures in siblings but NOT declared in parent. Clients can't call polymorphically.

Form 2: Unfactored Implementation

Identical method bodies in siblings. When logic changes, must update all copies.

Form 3: Both (Worst)

Signature AND body identical — e.g. Rectangle and Rectangle2D both define identical outcode() method.

💡

Key Example: Rectangle extends Rectangle2D, yet Rectangle ALSO defines outcode() with 20 lines of IDENTICAL code. Fix: delete from Rectangle, inherit from parent.

4. Wide Hierarchy

Superclass has excessive number of immediate children. Rule of thumb: >9 direct children = smell.

Symptoms

  • Hierarchy depth = 1–2, width > 15
  • Children at inconsistent abstraction levels
  • No clear grouping or organization
  • Java example: EventObject has 36 direct children

Example (Device System)

  • Start: Device with 3–4 children
  • After 18 months: 30+ direct children (FireAlarm, Camera, Sensor, SmartLight, etc.)
  • Fix: Introduce AlarmDevice, VideoDevice, SensorDevice as intermediate classes
  • Now Device has 3 children, not 30+
💡

Wide vs Unfactored: Unfactored = code duplication in siblings. Wide = too many immediate children, missing intermediate levels. Often BOTH occur together.

5. Speculative Hierarchy

Types in hierarchy added based on imagined future needs, not real requirements. Violates YAGNI.

Causes

  • Future-proofing — designing for all possible changes
  • Over-engineering beyond what requirements demand

Example

  • Code analyzer tool only needs HTML reports
  • Designer speculatively adds AnnotatedReport class (no requirement, no plan)
  • Fix: Collapse Hierarchy — remove AnnotatedReport (served no real purpose)
↓ Understandability ↓ Testability

6. Deep Hierarchy

Inheritance hierarchy is excessively deep. Rule of thumb: deeper than 6 levels.

Causes

  • Excessive focus on reuse — creating intermediate types just to enable reuse
  • Speculative generalization — adding supertypes for imagined needs

Problems

  • Method near root overridden many times → meaning lost at leaf
  • Field in supertype shadowed by local variable in subtype
  • Leaf behavior very hard to predict

Examples

  • SynthCheckBoxUI — long chain-like class hierarchy; leaf has 34 inherited methods; ButtonUI has NO members at all
  • Control center app — every physical device = separate class, new device = new subclass; leaf objects inherited hundreds of unused fields (pre-initialized arrays); each object = hundreds of KBs → thousands of objects → app ran out of memory. Fix: restructure intermediate types so leaf classes only inherit what they need
↓ Understandability ↓ Changeability ↓ Testability ↓ Reliability ↓ Performance

7. Rebellious Hierarchy

Subtype rejects methods provided by its supertype. Violates Liskov Substitution Principle (LSP).

How Rejection Manifests

Throw UnsupportedOperationException Empty/NOP method body Print "should not implement" Return error value

Causes

  • Concrete class becomes hierarchy root → subtypes forced to reject inherited methods they don't need
  • "Swiss-knife" supertype — provides everything, specialized subtypes reject irrelevant parts

Example (CallSite)

  • CallSite has setTarget() and getTarget()
  • ConstantCallSite inherits setTarget() but doesn't need it (rejects it)
  • Best Fix (Option 3): Introduce ChangeableCallSite as intermediate; MutableCallSite and VolatileCallSite extend it; setTarget() lives only there; ConstantCallSite no longer inherits it

Refactoring Options

  • Move method down — move to subtypes that actually need it
  • Remove method — if every subtype rejects it
  • Introduce intermediate type — create middle class for subtypes that support the method

Architectural Refactoring

PPT 5

Design Erosion

Gradual degradation of a software system's architecture due to continuous, unsystematic changes — causing a gap between intended design and actual implementation.

Consequences

Increased complexity Hard to modify/extend Higher bug risk Performance issues Expensive future rework

How to Avoid (Best Practices)

  • Avoid big bang approach; adopt iterative development
  • Continuously reassess the architecture
  • Refactor regularly; avoid ad hoc modifications
  • Don't treat design decisions as permanent
  • Prevent accumulation of technical debt

Shallow vs Deep Architectural Refactoring

AspectShallow RefactoringDeep Refactoring
Applied whenArchitecture exists only as models/viewsArchitecture already implemented (has code)
InvolvesModel refinement, modification of architectural designChanges to models AND actual implementation (code)
Code changesNo implementation/code changesCode refactoring along with architecture refactoring
ValidationArchitecture assessment methods, tools and software metricsTesting (ensure behavior unchanged), documentation, database schemas, reference architectures

Architectural Smells

  • Duplicate Design Artifacts — Same responsibility in multiple components. Example: Two services both implement their own user authentication instead of sharing an auth module.
  • Unclear Roles of Entities — Overlapping/poorly defined responsibilities. Example: UserManager and UserHandler doing similar things — developers confused about which to use.
  • Design Violations — Breaking established architectural rules. Example: UI layer directly accessing database instead of going through service layer.
  • Dependency Cycles — Components depend on each other circularly. Example: Service A → B → C → A
💡

Key reminder: All smells are suspicion, not proof. Always ask: "What is the system trying to achieve? Is this design choice justified?" — Smells = signals, not definitive defects.

📦 Case Study: Airbnb Architecture Evolution

PhaseArchitectureProblem
Phase 1Monolithic Ruby on Rails (Monorail)Scale issues as traffic grew
Phase 2Service-Oriented Architecture — Data Service, Business Logic, Presentation, API GatewayClear ownership ✅ but eventually more complexity
Phase 3 (2017–2020)Microservices Explosion — hundreds of small independent servicesHard to manage, complex dependencies, tracing requests difficult, team coordination overhead. Over-architecture!
Phase 4Hybrid (Micro + Macro Services) — related services merged into "macroservices"Solved: API unification, clearer boundaries, reduced dependencies

How Airbnb Refactored

  • Regrouped services — merged overly fragmented components into cohesive units
  • Redefined service boundaries — clear ownership and responsibilities
  • Reduced unnecessary dependencies
  • Balanced flexibility and simplicity

Simplifying Conditional Expressions

PPT 3

1. Decompose Conditional

Extract methods from the condition, then-part, and else-part. Make logic readable without inspecting implementation.
  • Complex conditionals increase method length, reduce clarity
  • Code shows what happens, not why — naming extracted methods exposes intent
  • Separate: the condition (decision) vs the actions (what happens)
// BEFORE
if (user.isActive() && !user.isBanned())
    access = true;
else
    access = false;
// AFTER
if (canAccess(user))
    access = grantAccess();
else
    access = denyAccess();

private boolean canAccess(User user) {
    return user.isActive() && !user.isBanned();
}

2. Consolidate Conditional Expression

Combine a sequence of conditional tests with the same result into a single conditional expression.
// BEFORE
double calculateLoan(User u) {
  if (u.income < 20000) return 0;
  if (u.creditScore < 600) return 0;
  if (u.hasCriminalRecord) return 0;
  if (u.isVIP) {
    if (u.yearsWithBank > 5)
      return u.income * 10;
  }
  return u.income * 5;
}
// AFTER
double calculateLoan(User u) {
  if (isHighRisk(u)) return 0;
  if (isPremiumEligible(u)) return u.income * 10;
  return u.income * 5;
}
boolean isHighRisk(User u) {
  return u.income < 20000 ||
         u.creditScore < 600 ||
         u.hasCriminalRecord;
}
boolean isPremiumEligible(User u) {
  return u.isVIP && u.yearsWithBank > 5;
}

3. Consolidate Duplicate Conditional Fragments

Same code in every branch of an if/else doesn't belong inside the conditional — move it outside.
  • If duplicate at the end → move after conditional
  • If duplicate at the beginning → move before
  • If in middle → check dependencies, reposition safely, extract method if multiple lines
// BEFORE
if (order.isExpress()) {
    shippingCost = 50;
    updateInventory(order); // duplicate
} else {
    shippingCost = 20;
    updateInventory(order); // duplicate
}
// AFTER
if (order.isExpress())
    shippingCost = 50;
else
    shippingCost = 20;

updateInventory(order); // moved outside

4. Remove Control Flag

Replace control flag variables (done, found, error) with break, continue, or return.

Approach 1: break/continue

// BEFORE
boolean fraudDetected = false;
for (Transaction t : transactions) {
  if (!fraudDetected) {
    if (t.amount > 10000) {
      fraudDetected = true;
    }
  }
}
// AFTER
for (Transaction t : transactions) {
  if (t.amount > 10000) {
    blockAccount(t.accountId);
    break; // direct exit
  }
}

Approach 2: return

// BEFORE
String error = "";
for (User u : users) {
  if (error.equals("")) {
    if (u.age < 18) error = "Underage";
  }
}
return error;
// AFTER
for (User u : users) {
  if (u.age < 18) return "Underage";
  if (!u.email.contains("@"))
    return "Invalid Email";
}
return "";

5. Replace Nested Conditional with Guard Clauses

Handle edge cases early with immediate exits to eliminate deep nesting. "If this rare case → exit immediately."
  • Nested if-else gives equal importance to all branches — misleading
  • Most branches are rare edge cases — guard clauses make this explicit
  • Reduces: indentation depth, cognitive load, hidden logic paths
  • Single exit point rule is NOT useful if it reduces clarity
// BEFORE (deeply nested)
double getPayAmount(Employee e) {
  double result;
  if (e.isActive()) {
    if (!e.isSuspended()) {
      if (!e.isOnLeave()) {
        result = e.getSalary();
      } else {
        result = leavePay(e);
      }
    } else {
      result = suspendedPay(e);
    }
  } else {
    result = 0;
  }
  return result;
}
// AFTER (guard clauses)
double getPayAmount(Employee e) {
  if (!e.isActive())    return 0;
  if (e.isSuspended())  return suspendedPay(e);
  if (e.isOnLeave())    return leavePay(e);
  return e.getSalary();
}

6. Replace Nested Conditional with Polymorphism

When conditional chooses different behavior based on object type → use polymorphism instead.
  • Avoids explicit switch/if-else on types
  • Simplifies extensions — just create a new subclass
  • Eliminates duplicate switch statements repeated across methods
// BEFORE: Switch on bird type
double getSpeed() {
  switch (_type) {
    case EUROPEAN: return getBaseSpeed();
    case AFRICAN:  return getBaseSpeed()
                       - getLoadFactor()
                       * _numberOfCoconuts;
    case NORWEGIAN_BLUE:
      return (_isNailed) ? 0 :
             getBaseSpeed(_voltage);
  }
}
// AFTER: Polymorphic
abstract class Bird {
    abstract double getSpeed();
}
class European extends Bird {
    double getSpeed() { return getBaseSpeed(); }
}
class African extends Bird {
    double getSpeed() {
        return getBaseSpeed() - getLoadFactor()
               * _numberOfCoconuts;
    }
}

7. Introduce Null Object

Return a special null object with default behavior instead of returning null. Eliminates repetitive null checks.
  • Null object implements same interface as real object but provides neutral/default behavior
  • Eliminates "annoying" duplicate null checks throughout client code
  • Increases system stability — no NullPointerExceptions

Example (Utility Site System)

// BEFORE: Repeated null checks
Customer c = site.getCustomer();
BillingPlan plan;
if (c == null) plan = BillingPlan.basic();
else plan = c.getPlan();
String name;
if (c == null) name = "occupant";
else name = c.getName();
int weeks;
if (c == null) weeks = 0;
else weeks = c.getHistory()
             .getWeeksDelinquent();
// AFTER: Null Object pattern
class NullCustomer extends Customer {
    public boolean isNull() { return true; }
    public String getName() { return "occupant"; }
    public BillingPlan getPlan() { return BillingPlan.basic(); }
    public PaymentHistory getHistory() {
        return PaymentHistory.newNull(); // also null object
    }
}
// Client code — clean, no null checks:
Customer c = site.getCustomer(); // returns NullCustomer if none
BillingPlan plan = c.getPlan();
String name = c.getName();
int weeks = c.getHistory().getWeeksDelinquent();

8. Introduce Assertion

Make implicit assumptions explicit using assertion statements. Assertion failure = programmer error (bug), not runtime error.
  • Communication Aid — tells future developers what state is required
  • Debugging Tool — catches bugs early
  • Performance Safety — assertions can be disabled in production
  • Avoid side effects — assertions should only check state, not change it
// BEFORE: Implicit assumption
double getExpenseLimit() {
    // Assumes one of these MUST be true
    // but never states it!
    return (_expenseLimit != NULL_EXPENSE) ?
        _expenseLimit :
        _primaryProject.getMemberExpenseLimit();
}
// AFTER: Explicit assertion
double getExpenseLimit() {
    Assert.isTrue(
        _expenseLimit != NULL_EXPENSE ||
        _primaryProject != null
    ); // Fails immediately if assumption broken!
    return (_expenseLimit != NULL_EXPENSE) ?
        _expenseLimit :
        _primaryProject.getMemberExpenseLimit();
}

Making Method Calls Simpler

PPT 3

Overview of Techniques

Rename Method

Name should communicate intention — think: what would the comment say?

Add Parameter

Method needs more info from caller. Watch out: long param lists are hard to remember.

Remove Parameter

Parameter no longer used. Delete only if not part of an interface.

Separate Query from Modifier

CQS Principle: A method either returns a value (query) OR changes state (command) — never both.

Parameterize Method

Multiple similar methods with different values → merge into one with parameter. Removes duplication.

Replace Parameter with Explicit Methods

Method runs different code based on enum param → provide separate methods instead. Adds compile-time checking.

Preserve Whole Object

Passing individual values from an object → pass the whole object. Future-proof, more readable.

Replace Parameter with Method

Receiver can compute the value itself → remove the parameter, let receiver call the method directly.

Introduce Parameter Object

Same group of params always travel together → create a class. Add logic to the object (e.g. range.includes(date)).

Remove Setting Method

Field should only be set at creation → remove setter, set in constructor, mark field final.

Hide Method

Method not used outside class → change to private. Improves encapsulation, reduces complexity.

Replace Constructor with Factory Method

When object creation needs flexibility — static factory method decides which object to create.

Encapsulate Downcast

Move cast inside the method — return concrete type, not Object. Caller code becomes cleaner.

Replace Error Code with Exception

Return value indicates error → throw exception instead. Separates normal logic from error handling.

Replace Exception with Test

Exception used for normal conditions → check condition upfront with if statement. Exceptions = exceptional situations only.

Replace Constructor with Factory Method — 3 Variants

Variant 1: Subclasses via Switch

✅ Subclasses hidden from clients, type-safe at compile time
❌ Must update switch for every new subclass

Variant 2: Dynamic Loading (Class.forName)

✅ No switch — add subclasses freely, no factory update needed
❌ Typos cause runtime errors; exposes subclass names as strings

Variant 3: Explicit Factory Methods

✅ Best for small stable set of subclasses; clear intent; strong type safety
❌ New factory method needed for each new subclass

Encapsulate Downcast

Move the cast inside the method instead of forcing client to perform it.
// BEFORE: Client forced to cast
Object lastReading() {
    return readings.lastElement();
}
// Caller:
Reading r = (Reading) site.lastReading();
// AFTER: Cast inside method
Reading lastReading() {
    return (Reading) readings.lastElement();
}
// Caller is clean:
Reading r = site.lastReading();

Replace Error Code with Exception

// BEFORE: Error code
int withdraw(int amount) {
    if (amount > _balance) return -1;
    else { _balance -= amount; return 0; }
}
// Caller:
if (account.withdraw(amount) == -1)
    handleOverdrawn();
// AFTER: Exception
void withdraw(int amount)
        throws BalanceException {
    if (amount > _balance)
        throw new BalanceException();
    _balance -= amount;
}
// Caller:
try {
    account.withdraw(amount);
    doTheUsualThing();
} catch (BalanceException e) {
    handleOverdrawn();
}

Replace Exception with Test — Resource Pool Example

Exceptions should only signal unexpected/exceptional situations, not normal program flow. Check condition first with if statement.
// BEFORE: Exception for normal flow
Resource getResource() {
    try {
        result = (Resource) available.pop();
        allocated.push(result);
        return result;
    } catch (EmptyStackException e) {
        result = new Resource();
        allocated.push(result);
        return result;
    }
}
// AFTER: Test first
Resource getResource() {
    if (available.isEmpty())
        result = new Resource();
    else
        result = (Resource) available.pop();
    allocated.push(result);
    return result;
}

Dealing with Generalization

PPT 3

What is Generalization?

Moving functionality up or down a class hierarchy to eliminate duplication and ensure each piece of logic is in the most appropriate class.

1. Pull Up Field

Two subclasses have the same field → move field to superclass.

Example (Digital Payments): BankTransfer and ScannerPayment both have transactionID field → pull up to Payment superclass. Now history log can use Payment type uniformly.

2. Pull Up Method

Methods with identical results in multiple subclasses → move to superclass.

Example (Food Delivery): VegOrder and NonVegOrder both have identical calculateDeliveryFee() → if base fee changes, must update both. Fix: move to Order superclass.

3. Pull Up Constructor Body

Subclasses have constructors with mostly identical bodies → create superclass constructor, call super().

Example (Government ID): Resident and NRI profiles both verify fullName and DOB at creation → put in Citizen superclass constructor. Subclasses call super(name, dob) and handle specifics (like Visa status).

4. Push Down Method

Behavior in superclass is only used by some subclasses → move down.

Example (Banking): BankAccount has calculateEarlyWithdrawalPenalty() but SavingsAccount doesn't need it → push down to FixedDepositAccount only.

5. Push Down Field

Field in superclass only used by some subclasses → move field down.

Example (Indian Railways): Train superclass has acTierCount but UnreservedExpress doesn't need it (always null) → push down to only the subclasses that have AC tiers.

6. Extract Subclass

A class has features used by only some instances → create a subclass.

Example (E-commerce): Product has warrantyPeriod and isReturnable but for groceries these are empty. Fix: extract ElectronicProduct subclass for warranty data.

7. Extract Superclass

Two classes have similar features → create a superclass, move common features into it.

Example (Educational Institutes): Student has id, name, getFees() and Professor has id, name, getSalary(). Extract MemberOfInstitute superclass. Now campus gate system just checks for MemberOfInstitute.

8. Extract Interface

Decouple a class from clients by moving a subset of public methods into a new interface.

Example: Timesheet depends on full Employee but only needs getRate() and hasSpecialSkill(). Extract Billable interface. Now Computer can also implement Billable and be used in Timesheet.

// Before: double charge(Employee emp, int days)
// After: double charge(Billable emp, int days)
// Now Computer implements Billable too!

9. Collapse Hierarchy

A superclass and its subclass are no longer significantly different → merge them into a single class.
  • Pick which class to remove (parent or child)
  • Move all fields/methods to the class you're keeping
  • Update all references to the removed class
  • Delete the empty, unused class

Example: Employee (parent) and Salesman (child). Salesman has no unique behavior → move remaining Salesman code into Employee, update all usages, delete Salesman.

10. Form Template Method

Common steps moved to superclass, subclasses handle differing parts. Superclass defines template (order of steps); subclasses override specific steps.
  • Remove duplication: identical steps in superclass, different formatting in subclasses
  • Easy to extend: just create new subclass with different step implementations

Example (Statement Formatting)

  • Customer.statement() (ASCII) and htmlStatement() (HTML) do same steps but different formatting
  • Create TextStatement and HtmlStatement subclasses
  • Both have value() method with same steps calling headerString(), eachRentalString(), footerString()
  • Pull value() up to Statement superclass; declare header/detail/footer as abstract
  • Result: superclass controls workflow, subclasses define formatting only

11. Replace Inheritance with Delegation

Subclass only needs part of superclass → keep a reference to the superclass and delegate calls instead of inheriting.
  • Clarify intent — shows subclass only uses part of superclass
  • Control interface — expose only needed methods
  • Avoid inheriting unnecessary fields or unsuitable methods

Example (Stack inheriting Vector)

  • Stack extends Vector → gets many unwanted methods (insertElementAt, etc.)
  • Fix: Create private Vector _vector = new Vector(); inside Stack
  • Delegate push/pop to _vector; expose only size() and isEmpty() as needed
  • Remove extends Vector; Stack only exposes relevant operations

12. Replace Delegation with Inheritance

Class delegates most of its work to another class → just inherit from it to avoid many forwarding methods.
  • Reduce boilerplate — avoid writing many simple forwarding methods
  • Simpler interface — automatically get all methods by inheriting
⚠️

When NOT to use: If you only need SOME methods (partial use) — don't inherit. Also avoid if delegate object is shared and mutable.

Example (Employee → Person)

  • Employee has Person _person field and wraps ALL Person methods
  • Fix: class Employee extends Person; remove all wrapper methods; remove _person field
  • Now Employee directly inherits all Person functionality
