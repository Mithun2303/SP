Creational Patterns

Object Creation Mechanisms

Abstract the instantiation process so systems are independent of how objects are created, composed, and represented.

2.1.1

Singleton

"Ensures a class has only one instance and provides a global access point to it."

Intent

  • Restrict instantiation to exactly one object
  • Provide a single shared global access point

When to Use

  • Log files, Database connections, Caching
  • Configuration settings, Device drivers
  • One coordinator object needed system-wide

Key Participants

  • Singleton — private constructor + static get_instance()

Benefits & Limitations

Benefits
  • Prevents duplicate object creation
  • Easy cross-component sharing
  • Controls resource usage (memory, DB, threads)
  • Lazy initialisation supported
Limitations
  • Global state — acts like a global variable, harder to test
  • Hidden dependencies — breaks modular design
  • Multithreading requires special thread-safety care
  • Private constructor blocks inheritance
One InstanceLoggerDB PoolConfig
2.1.2

Abstract Factory

aka "Factory of Factories"
"Provides an interface for creating families of related objects without specifying their concrete classes."

Intent

  • Create families of related objects guaranteed to work together
  • Enforce consistency — products from same family are compatible
  • Hide creation logic from client code

When to Use

  • Need multiple product families (Electric/Petrol vehicles)
  • System configurable by platform (Win/Mac/Linux UI widgets)
  • Bank accounts + card types must match per tier

Key Participants

  • AbstractFactory — interface declaring create methods
  • ConcreteFactory — creates one product family
  • AbstractProduct — interface per product type (Car, Bike)
  • ConcreteProduct — specific instance (ElectricCar)
  • Client — uses only abstract interfaces

Benefits & Limitations

Benefits
  • Isolates concrete classes from client
  • Makes exchanging product families easy
  • Promotes consistency among products in the same family
Limitations
  • Hard to support new product types — must change all factories
  • Increased number of classes and interfaces
  • Can be overkill for simple scenarios
vs Factory MethodFactory Method creates one product; Abstract Factory creates families of related products.
Product FamilyOS WidgetsVehicle Plant
2.1.3

Factory Method

aka "Virtual Constructor"
"Defines an interface for creating an object, but lets subclasses decide which class to instantiate."

Intent

  • Avoid tight coupling between client and concrete classes
  • Open/Closed Principle — extend without modifying
  • Vary object creation without changing client code

When to Use

  • Exact type not known until runtime
  • Notification system: SMS / Email / Push
  • Payment: Card / UPI / Net Banking

Key Participants

  • Creator — abstract class declaring the factory method
  • ConcreteCreator — overrides to return specific product
  • Product — interface for created objects
  • ConcreteProduct — specific implementation

Benefits & Limitations

Benefits
  • Decouples object creation from usage
  • Supports polymorphism and scalability
  • Enables code reuse and testability
Limitations
  • More classes and complexity than simple instantiation
  • Overhead of subclassing just to change object creation
Subclass DecidesNotificationsPayment
2.1.4

Prototype

"Creates new objects by cloning existing ones — avoiding costly re-initialization."

Intent

  • Avoid expensive object creation from scratch
  • Clone a prototype instead of using new
  • Hide complex construction logic from client

When to Use

  • Object creation is expensive (DB / network / parsing)
  • Need many similar objects with slight variations
  • Duplicate invoice templates, covering letters

Key Participants

  • Prototype — interface with clone()
  • ConcretePrototype — implements clone (shallow/deep copy)
  • Client — calls clone() instead of new

Benefits & Limitations

Benefits
  • Fast creation — avoids time-consuming constructors
  • Reduces code duplication — centralised prototype logic
  • Runtime flexibility — clone and modify at runtime
Limitations
  • Deep cloning can be hard and error-prone
  • Hidden side effects if shared references aren't handled
  • All classes must support a clone() interface
vs Factory MethodFactory uses inheritance; Prototype uses composition (clone). Use Prototype when duplicating object state matters.
Clonecopy.deepcopy()Document Template
2.1.5

Builder

"Separates construction of a complex object from its representation — same process, different representations."

Intent

  • Construct complex objects step by step
  • Avoid telescoping constructors (too many parameters)
  • Create different representations using the same building process

When to Use

  • Many optional components / configurations
  • Step-by-step creation required
  • Pizza ordering, Car manufacturing, CV/Resume builder

Key Participants

  • Builder — abstract interface for building steps
  • ConcreteBuilder — implements steps (LuxuryVacation, BudgetVacation)
  • Director — controls build order (TravelAgency)
  • Product — the complex object being built

Benefits & Limitations

Benefits
  • Better control over construction process step-by-step
  • Readability — eliminates constructor parameter confusion
  • Supports different representations via same process
  • Builders can be reused for various products
Limitations
  • Requires multiple extra classes (Builder, ConcreteBuilder, Director)
  • May be overkill for simple object construction
Step-by-StepDirectorPizza / CarImmutable

Structural Patterns

Composing Objects & Classes

Concerned with how objects and classes are assembled into larger structures — while keeping them flexible and efficient.

2.2.1

Adapter

aka "Wrapper"
"Converts the interface of a class into another interface the client expects."

Intent

  • Let incompatible classes work together
  • Reuse existing / third-party code without modification

When to Use

  • Integrating legacy code into a new system
  • Payment gateway: UPI/Paytm/Card → common interface
  • Media player supporting additional formats

Key Participants

  • Target — interface client expects
  • Adaptee — existing incompatible class
  • Adapter — wraps Adaptee, implements Target
  • Client — uses only the Target interface

Benefits & Limitations

Benefits
  • Reuses existing code without modifying it
  • Improves compatibility across systems
  • Clients remain decoupled from third-party APIs
  • Makes the system extensible and easy to test/mock
Limitations
  • Increased class count — can lead to class explosion
  • Added complexity via extra abstraction layer
  • Not ideal for frequently changing interfaces
vs FacadeAdapter makes incompatible interfaces compatible; Facade simplifies a subsystem. Adapter is retroactive; Facade is designed upfront.
CompatibilityWrapperLegacy
2.2.2

Bridge

"Separates an object's abstraction from its implementation so both can vary independently."

Intent

  • Decouple abstraction from implementation — two separate hierarchies
  • Prevent class explosion from combining multiple dimensions
  • Switch implementations at runtime

When to Use

  • Permutation explosion: TV/Radio × Basic/Advanced Remote
  • Abstraction and implementation must evolve independently
  • Smart home: appliances × switch types

Key Participants

  • Abstraction — high-level control (RemoteControl)
  • RefinedAbstraction — extended control (AdvancedRemote)
  • Implementor — device interface (Device)
  • ConcreteImplementor — TV, Radio

Benefits & Limitations

Benefits
  • Decouples abstraction from implementation
  • Avoids class explosion — each hierarchy extended independently
  • Improves flexibility and scalability
Limitations
  • Increased complexity due to additional abstraction layers
  • Performance overhead due to indirection between hierarchies
Two HierarchiesDevice + RemoteRuntime Switch
2.2.3

Composite

"Composes objects into tree structures to represent part-whole hierarchies — treat individual objects and groups uniformly."

Intent

  • Model hierarchical tree structures
  • Uniform treatment of leaves and composites
  • Avoid if/else to distinguish individual vs collection

When to Use

  • File system: file vs folder
  • Org chart: employee vs manager with team
  • UI components, DOM elements (HTML/XML)

Key Participants

  • Component — common interface (show_info)
  • Leaf — individual object (Developer, Tester)
  • Composite — holds children, delegates operations (Manager)
  • Client — treats all via Component interface

Benefits & Limitations

Benefits
  • Easy to model nested/tree data (folders, UI, org charts)
  • Uniform interface — treat leaf and composite the same
  • Open/Closed Principle — easily add new component types
  • Simplified client code — no conditional statements
Limitations
  • Harder to type-check strictly — leaf vs composite look the same
  • Complex recursive structures can lead to overdesign
Tree StructureLeaf + CompositeFile System
2.2.4

Decorator

"Attaches new behaviors to objects at runtime by wrapping them in decorator objects."

Intent

  • Add behavior dynamically without subclassing
  • Avoid subclass explosion for feature combinations
  • Follow Open/Closed Principle

When to Use

  • Text Editor: Bold + Italic + Underline (mix-and-match)
  • E-commerce: RAM + SSD + GPU add-ons
  • Video streaming: subtitles + audio + quality options

Key Participants

  • Component — interface for objects that can be wrapped
  • ConcreteComponent — base object (PlainEditor)
  • BaseDecorator — wraps component, delegates calls
  • ConcreteDecorator — adds specific feature (BoldDecorator)

Benefits & Limitations

Benefits
  • Adds new behavior without modifying existing classes
  • Avoids subclass explosion for combinations of features
  • Follows Open/Closed Principle
Limitations
  • Too many small decorator classes can clutter the design
  • Deeply nested decorators make debugging tricky
  • Not suitable when global behavior changes are needed
vs ProxyBoth wrap objects; Decorator adds features; Proxy controls access. Decorator is client-driven; Proxy may hide the real object entirely.
Runtime WrapBold+ItalicDynamic Behavior
2.2.5

Facade

"Provides a unified, high-level interface to a complex subsystem — hiding complexity behind a simple face."

Intent

  • Simplify complex subsystem usage
  • Decouple client from internal workings
  • Encapsulate complexities into a single entry point

When to Use

  • Bank account opening: ID verify + credit check + create + welcome kit
  • E-commerce order: availability + payment + inventory + fulfil
  • Travel booking: flight + hotel + car in one call

Key Participants

  • Facade — simplified interface (BankAccountFacade)
  • Subsystems — complex internal classes (IDVerifier, CreditChecker…)
  • Client — calls only the facade

Benefits & Limitations

Benefits
  • Simplifies complex subsystems for clients
  • Reduces client-subsystem dependencies
  • Promotes loose coupling and layered architecture
  • Improved readability and maintainability
Limitations
  • Not suitable when full control over subsystems is required
  • Overkill if the subsystem is already simple
  • Facade can become a "god object" if overloaded
SimplifyOne Entry PointDecouple
2.2.6

Flyweight

"Reduces memory by sharing intrinsic (common) state between many similar objects — unique extrinsic state passed by client."

Intent

  • Minimize memory by sharing intrinsic state
  • Improve performance — reduce object creation overhead
  • Scale efficiently with large numbers of similar objects

Intrinsic vs Extrinsic

  • Intrinsic — shared, stored in flyweight (tree type: Oak, color, texture)
  • Extrinsic — unique per instance, passed by client (tree position x, y)

When to Use

  • Thousands of trees in a game (shared type, unique position)
  • Text editor: font/size/color shared across same characters
  • DB connection pools, caching systems

Key Participants

  • Flyweight — shared object (TreeType)
  • FlyweightFactory — creates and caches flyweights
  • Context — holds extrinsic state (Tree with x, y)

Benefits & Limitations

Benefits
  • Massive memory savings for large numbers of similar objects
  • Speeds up object creation by reusing shared state
  • Encourages reuse and object sharing
Limitations
  • Adds complexity if not actually needed
  • Requires careful separation of intrinsic vs extrinsic state
  • Extrinsic state management adds overhead to calling code
vs SingletonSingleton = one instance total. Flyweight = one instance per type, many types allowed.
Share StateIntrinsic/ExtrinsicMemory
2.2.7

Proxy

aka "Surrogate"
"Provides a surrogate that controls access to another object — adding security, lazy loading, caching, or logging."

Intent

  • Control access to a sensitive or expensive object
  • Lazy initialization — create only when needed
  • Add logging / caching / monitoring without changing real object

Types of Proxy

  • Remote — represents object on a different server
  • Virtual — delays heavy creation (image loading)
  • Protection — access control by role (Admin vs Guest)
  • Smart — adds logging, caching, monitoring

Key Participants

  • Subject — common interface
  • RealSubject — actual object
  • Proxy — controls access, same interface as RealSubject

Benefits & Limitations

Benefits
  • Adds controlled access to the real object
  • Supports lazy loading, security, logging, and caching
  • Reduces resource consumption for heavy objects
Limitations
  • Adds an extra level of indirection
  • Can introduce performance overhead
  • Response delay if proxy involves network/remote
Control AccessLazy LoadSecurityCache

Behavioural Patterns

Communication Between Objects

Concerned with algorithms and the assignment of responsibilities — defining how objects interact and distribute work.

2.3.1

Command

"Turns a request into a standalone object — enabling undo/redo, queuing, logging, and decoupled execution."

Intent

  • Encapsulate requests as objects
  • Decouple sender (invoker) from receiver
  • Support undo/redo, queue, schedule, and log

When to Use

  • GUI buttons / menu actions
  • Job schedulers: DataBackup, ReportGen, EmailSend
  • Database transactions with rollback
  • Remote control buttons

Key Participants

  • Command — interface with execute()
  • ConcreteCommand — binds action to receiver
  • Invoker — triggers command (JobScheduler)
  • Receiver — performs actual business logic
  • Client — creates and configures commands

Benefits & Limitations

Benefits
  • Decouples sender (invoker) and receiver completely
  • Easy to add new commands without changing existing code
  • Supports undo, redo, logging, macros
  • Can queue or schedule commands
Limitations
  • Increased class complexity — one class per command
  • Memory and performance overhead for large command histories
Encapsulate RequestUndo/RedoQueue
2.3.2

Interpreter

"Defines a grammar for a language and provides an interpreter to evaluate sentences in that language."

Intent

  • Parse and evaluate expressions (math, SQL, regex)
  • Implement simple scripting / DSL (Domain Specific Language)
  • Build expression trees recursively

When to Use

  • SQL parsers, Regular expression engines
  • Business rule engines, Math expression evaluators
  • Simple search/filter logic (AND, OR, NOT)

Key Participants

  • AbstractExpression — interface with interpret()
  • TerminalExpression — leaf node (Number)
  • NonTerminalExpression — compound node (Add, Subtract)
  • Context — global info for interpreter
  • Client — builds and triggers expression tree

Benefits & Limitations

Benefits
  • Good for implementing simple languages and expression trees
  • Encapsulates parsing and evaluation logic cleanly
  • Supports recursion naturally (expressions within expressions)
Limitations
  • Becomes complex and inefficient for large grammars
  • Better suited for small DSLs only — not production parsers
  • Maintenance difficult as grammar grows
Grammar / DSLExpression TreeSQL / Regex
2.3.3

Iterator

"Provides sequential access to elements of a collection without exposing its internal representation."

Intent

  • Traverse collections uniformly regardless of internal structure
  • Support multiple simultaneous traversals
  • Unify access to arrays, lists, trees, etc.

When to Use

  • Music playlist (hide internal list type)
  • File system traversal, Graph DFS/BFS
  • Social media feeds, DB query result processing

Key Participants

  • Iterator — interface (has_next, next)
  • ConcreteIterator — implements traversal (SongIterator)
  • Aggregate — collection interface (create_iterator)
  • ConcreteAggregate — actual collection (Playlist)

Benefits & Limitations

Benefits
  • Decouples client from concrete collection implementation
  • Flexible traversal — multiple strategies (forward, backward, DFS)
  • Concurrent iteration on the same collection
  • Encapsulates iteration logic outside the collection
Limitations
  • Modifying the collection during traversal is risky
  • Overkill / unnecessary complexity for simple collections
Sequential Accesshas_next/nextPlaylist
2.3.4

Mediator

"Centralizes communication between multiple objects — preventing direct dependencies and reducing coupling from O(n²) to O(n)."

Intent

  • Objects communicate via mediator, not directly
  • Centralize control and coordination logic

When to Use

  • Chat application (ChatRoom routes messages)
  • Air traffic control tower
  • GUI frameworks (buttons/fields coordinated through a controller)

Key Participants

  • Mediator — interface (send_message)
  • ConcreteMediator — ChatRoom manages users
  • Colleague — User objects that only talk to mediator

Benefits & Limitations

Benefits
  • Loose coupling — avoids direct dependencies between colleagues
  • Improved maintainability and reusability
  • Simplified testing of individual components
Limitations
  • Mediator can become a "God Object" — too much logic in one place
  • Single point of failure
  • Transparency and traceability becomes difficult
  • Performance overhead for high-frequency messaging
Central HubChat RoomLoose Coupling
2.3.5

Memento

"Captures and stores an object's internal state so it can be restored later — without exposing implementation details."

Intent

  • Snapshot/restore state without breaking encapsulation
  • Undo/redo, checkpoints, rollback mechanisms

When to Use

  • Text editor undo/redo
  • Game save/restore points
  • Database transaction rollback, Version control

Key Participants

  • Originator — object whose state is saved (TextEditor)
  • Memento — stores the state snapshot (EditorState)
  • Caretaker — manages mementos, triggers undo (UndoManager)

Benefits & Limitations

Benefits
  • Preserves encapsulation — state saved without exposing internals
  • Structured approach for undo/redo features
  • Flexible state management at multiple points
  • Simplifies testing of state transitions
Limitations
  • Memory overhead — storing many snapshots is expensive
  • Added complexity managing the caretaker
  • State consistency issues if Originator structure changes
  • Performance overhead for frequent or large state saves
SnapshotUndo/RedoRollback
2.3.6

Observer

aka "Publish-Subscribe"
"One-to-many dependency: when a subject changes state, all registered observers are automatically notified."

Intent

  • Auto-notify multiple dependents on state change
  • Loose coupling between subject and observers
  • Add/remove observers at runtime

When to Use

  • Stock price → notifies mobile app + trading screen + email
  • GUI event handling, news feeds
  • Weather monitoring, social media notifications

Key Participants

  • Observer — interface (update)
  • Subject/Publisher — maintains observer list, notifies (Stock)
  • ConcreteObserver — reacts to updates (MobileApp, EmailAlert)

Benefits & Limitations

Benefits
  • Loose coupling — subject and observers are independent
  • Scalable — easily add or remove observers at runtime
  • Suitable for event-driven systems
  • Observers reusable in different contexts
Limitations
  • Memory leaks if observers are not properly removed
  • Performance overhead when notifying many observers
  • Observers may receive unintended/unnecessary updates
  • Order of notification is not guaranteed
One-to-ManyPub/SubEvent Driven
2.3.7

State

"Allows an object to change its behavior when its internal state changes — as if the object changed its class."

Intent

  • Each state is its own class — eliminates if/else explosion
  • Object delegates behavior to current state object
  • Implement finite state machines (FSM) cleanly

When to Use

  • Vending machine: No money → Has money → Dispense
  • Traffic lights: Stop → Ready → Go → Blinking
  • Media player: Playing, Paused, Stopped
  • Game character: Idle, Running, Attacking

Key Participants

  • State — interface for all states
  • ConcreteState — behavior for one state (NoMoneyState)
  • Context — holds current state reference (VendingMachine)

Benefits & Limitations

Benefits
  • Eliminates complex conditional logic (if/else / switch)
  • Improved code maintainability
  • Adheres to Open/Closed Principle — add new states easily
  • Simplifies transitions, enhanced testability
Limitations
  • Increased number of classes — one per state
  • Potential for complexity if not well-designed
  • Memory overhead for many state objects
vs StrategyState: transitions driven internally; states know each other. Strategy: algorithm chosen explicitly by client; strategies are independent.
FSMNo if/elseVending Machine
2.3.8

Strategy

"Defines a family of algorithms, encapsulates each one, and makes them interchangeable at runtime."

Intent

  • Select algorithm at runtime without if/else
  • Isolate algorithm details from context
  • Open/Closed — add new strategies without changing context

When to Use

  • Navigation: Shortest / Fastest / Scenic route
  • Sorting: QuickSort vs MergeSort based on data
  • Payment: Card / UPI / Wallet
  • Data compression: ZIP vs GZIP

Key Participants

  • Strategy — interface (calculate_route)
  • ConcreteStrategy — one algorithm (ShortestRoute)
  • Context — holds strategy reference, delegates to it (Navigator)

Benefits & Limitations

Benefits
  • Flexibility — swap algorithm at runtime
  • Eliminates conditional logic
  • Decouples algorithm from the context class
  • Testability and Open/Closed Principle
Limitations
  • Increased number of classes — one per strategy
  • Client must be aware of available strategies
  • Overhead if only a few simple algorithms are used
Swap AlgorithmNavigationPayment
2.3.9

Template Method

"Defines the skeleton of an algorithm in a base class, letting subclasses override specific steps without changing the overall structure."

Intent

  • Fix the algorithm structure — vary only specific steps
  • Avoid code duplication — common code in one place
  • Enforce fixed order of execution

When to Use

  • Data processing: Read → Validate → Process → Save (CSV vs JSON differ in how)
  • Online payment: Authenticate → Process → Receipt
  • Report generation: template fixed, data source varies

Key Participants

  • AbstractClass — template_method() (final) + abstract hook steps
  • ConcreteClass — implements abstract steps (CSVProcessor, JSONProcessor)

Benefits & Limitations

Benefits
  • Code reusability — common steps defined once
  • Consistent algorithm structure enforced across subclasses
  • Separation of concerns
  • Improved maintainability and extensibility
Limitations
  • Tight coupling via inheritance — subclasses depend on base class
  • Increased complexity — more classes needed
  • Template method can become bloated with too many hooks
vs StrategyTemplate: uses inheritance, varies steps at compile time. Strategy: uses composition, swaps entire algorithm at runtime.
Algorithm SkeletonHook MethodsCSV/JSON
2.3.10

Visitor

"Lets you add new operations to an existing object structure without modifying the classes — operations live in separate visitor objects."

Intent

  • Add operations without changing element classes
  • When structure is stable but operations grow over time
  • Centralize related operations across element types

When to Use

  • Tax: Electronics/Clothing/Groceries × GST/Import Duty
  • Document: Paragraphs/Images/Tables × SpellCheck/Export
  • Insurance: Car/Home/Health policy × Risk/Premium calc

Key Participants

  • Visitor — interface with visit(ElementType) per type
  • ConcreteVisitor — implements operations (TaxCalculator)
  • Element — interface with accept(visitor)
  • ConcreteElement — calls visitor.visit(self)

Double Dispatch

  • Operation resolved by both element type and visitor type

Benefits & Limitations

Benefits
  • Separation of concerns — operations separated from data
  • Easy to add new operations without modifying elements
  • Centralized logic, reusability
  • Double dispatch handles both type dimensions
Limitations
  • Adding new element types requires all visitors to change
  • Visitors may need to access internal element state — violates encapsulation
  • Overall complexity increases with many element types
Add OperationsDouble DispatchTax Calc
2.3.11

Chain of Responsibility

aka "CoR"
"Passes a request along a chain of handlers — each decides to handle or forward — decoupling sender from receiver."

Intent

  • Multiple handlers can process a request
  • Sender doesn't know which handler will act
  • Add/remove/reorder handlers at runtime

When to Use

  • Customer support: L1 → L2 → Manager escalation
  • ATM: handle different note denominations
  • HTTP Middleware: auth → rate limit → route
  • Logging: DEBUG → INFO → WARNING → ERROR

Key Participants

  • Handler — interface (set_next, handle)
  • ConcreteHandler — handles or calls next (Level1Support)
  • Client — builds chain and sends requests

Benefits & Limitations

Benefits
  • Loose coupling — sender doesn't need to know the receiver
  • Enhanced flexibility — add/remove handlers easily
  • Improved maintainability — centralised handler logic
  • Reduces code duplication via shared handler behaviour
Limitations
  • Increased complexity with many handlers
  • Performance overhead — request may traverse entire chain
  • Debugging is difficult — hard to trace the flow
  • Handler order is crucial — wrong order breaks logic
  • Risk of unhandled requests if chain is incomplete
Pass the BuckMiddlewareEscalation
